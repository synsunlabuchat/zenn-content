先月、うちのチーム（4人のエンジニア、スタートアップ）で顧客問い合わせメールを自動分類するシステムを作ることになった。カテゴリは8種類、メールは英語・日本語・スペイン語が混在するという割とカオスな条件。最初に書いたプロンプトは本当にひどくて、精度が62%しか出なかった。

単純な指示文を書いてモデルに投げる。返ってくる分類が明らかにおかしい。「もっと丁寧に書けばいいんじゃない?」と同僚に言われて、「それはそうなんだけど…」ってなった。丁寧に書くとはどういうことか、具体的に何をすれば精度が上がるのか。その答えを探して2週間、Chain-of-ThoughtとFew-Shotをひたすら実験した。その記録を書く。

---

## 「考えながら答えさせる」だけで精度が10%上がった理由

Chain-of-Thought（以下CoT）の基本的な考え方は有名だと思う。「答えを出す前に推論過程を書かせる」というやつ。でも最初、正直なめていた。「ただ考える手順を書かせるだけで何が変わるの?」と思っていた。

変わった。かなり。

うちの分類タスクで試したとき、CoTなしで62%だった精度がCoTありで73%になった。同じモデル（claude-claude-sonnet-4-6、2026年2月時点）、同じデータで。

なぜ効くのか、自分なりに理解したことを書く。モデルは左から右にトークンを生成していく。答えを直接出力させると、そのトークンを生成する時点でモデルが参照できるコンテキストは入力だけ。でも推論ステップを先に書かせると、その推論テキスト自体がコンテキストになる。つまりモデルは自分の「思考メモ」を見ながら最終答えを出せる。人間がメモを取りながら考えるのと似たような話だ。

実際にうちで使ったプロンプトの骨格はこんな感じ:

```python
SYSTEM_PROMPT = """
あなたは顧客問い合わせ分類の専門家です。
メールを分析し、以下のカテゴリに分類してください:
[請求, 技術サポート, 返品, 配送, アカウント, 機能要望, クレーム, その他]

重要: 最終的なカテゴリを答える前に、必ず以下のステップで分析を行ってください:
1. メールの主な訴えや要求を1〜2文で要約する
2. キーワードや文脈から示唆されるカテゴリ候補を列挙する
3. 候補の中から最も適切なものを選んだ理由を説明する
4. 最終カテゴリを出力する

出力形式:
分析: [推論ステップ]
カテゴリ: [最終カテゴリ名]
"""
```

ここで気づいたことが一つある。「考えてから答えてください」とだけ言うよりも、**考えるステップを具体的に番号で指定した方が効果が高かった**。抽象的な指示を与えると、モデルは「分析しました」みたいな一行を書いてすぐ答えを出してしまうことがある。ステップを明示すると、ちゃんとそれぞれのステップを踏む。

一つ注意点として、CoTは出力トークンが増える分、コストも増える。うちのケースでは1リクエストあたりの出力トークンが約3倍になった。コストとのトレードオフは意識した方がいい。バッチ処理ならともかく、レイテンシが重要なリアルタイム用途には向かないことも多い。

## Few-Shotの誤解 — 例を増やせばいいわけじゃない

Few-Shot Promptingも基本的な概念は知っていた。例をいくつか見せて、パターンを学習させる。でもここで盛大に失敗した。

最初にやったのは、8カテゴリそれぞれから5例ずつ、合計40例をプロンプトに詰め込むことだった。コンテキストが長くなって、精度は…下がった。65%。CoTなしの最初より悪い。

何が起きたかというと、例が多すぎてモデルが混乱したのと、例の質がバラバラだったのが問題だった。「とりあえず量で押せ」は完全に間違いだった。

正しいFew-Shotの使い方について、2週間で学んだことをまとめる。

**1. 例の「難しさ」を意図的に選ぶ**

明らかな例を大量に入れても意味がない。「これは絶対に技術サポートだよね」みたいな簡単なケースは、そもそも0-shotでも正解できる。Few-Shotが効くのはエッジケースだ。うちの場合、「請求」と「クレーム」が混在するメール（請求内容に怒っているケース）とか、「返品」と「配送」が区別しにくいケースとかを意図的に選んだ。

**2. 例はカテゴリごとに均等でなくていい**

実際のデータ分布に近い比率にした方がいい場面もあれば、苦手なカテゴリに偏らせた方がいい場面もある。うちは「その他」カテゴリの誤分類が多かったので、「その他」の例を多めにした。

**3. 例の出力フォーマットがCoTと一致していること**

これが一番重要かもしれない。CoTと組み合わせる場合、Few-Shotの例にもちゃんと推論ステップを含めないとダメだ。例が「INPUT → カテゴリ: X」という形式で、本番では「INPUT → 分析: ... → カテゴリ: X」という形式を期待しても、モデルは例のフォーマットに引っ張られる。

実際の例の構造はこういう感じにした:

```
例1:
メール: 「先月の請求書に見覚えのない金額が含まれています。2月分で通常より5,000円高い。詐欺ではないかと心配しています。」

分析:
1. 要約: 請求書の金額に納得できず、不正請求の可能性を疑っている
2. 候補カテゴリ: 請求（金額の不一致）、クレーム（怒り・不満の感情）
3. 理由: 感情的な表現はあるが、主訴は請求内容の確認・説明であり、解決策は請求部門への問い合わせ。怒りはクレームの副次的要素に過ぎない。
カテゴリ: 請求
```

## CoTとFew-Shotを組み合わせたときに起きた予想外のこと

個別に効果を確認してから両方を組み合わせたら、さらに精度が上がって最終的に81%になった。ただ、組み合わせる過程でいくつか予想外の問題が出た。

**「例の推論」をコピーし始める問題**

Few-Shotの例に含めた推論の癖やフレーズを、モデルが過度にコピーするようになった。たとえば、例の中で「〜と考えられる」という表現を多用したら、全ての推論で同じ表現を使い始めた。これ自体は致命的じゃないけど、推論の多様性が下がって、例に含まれないケースへの汎化が若干落ちた気がする（定量的に確認はしていない）。

対策として、例の表現をできるだけ多様にするようにした。同じ構造でも語彙や言い回しを変える。

**Few-Shotの例の順序が意外と影響する**

これは本当に驚いた。同じ例を使っても、プロンプト内の並び順で精度が変わる。具体的には、プロンプトの末尾に近い例ほど影響が強い傾向がある（recency bias）。だから一番難しいエッジケースの例を最後に置くようにした。100%理論的に正しいかは自信がないけど、うちのデータでは再現性があった。

**システムプロンプトに入れるべきか、ユーザーターンに入れるべきか**

Few-Shotの例をどこに入れるかで迷った。Claude APIの場合、システムプロンプトに入れるのとユーザーターンに入れるのとで挙動が微妙に違う。うちは最終的に、カテゴリの定義と指示はシステムプロンプト、Few-Shotの例はユーザーターンに入れる形に落ち着いた。理由は、例をキャッシュしやすくするためだ（Claudeにはprompt cachingの機能がある）。40例を毎リクエスト送信していたら、コストが馬鹿にならない。

## 失敗から学んだ: Few-Shotの例を選ぶ作業を甘く見ていた

正直に言う。一番時間がかかったのも、一番重要だったのも、**良い例を選ぶ作業そのもの**だった。プロンプトの文言を工夫することより、よっぽど重要だった。

最初の失敗例は「自分が直感的にわかりやすいと思った例」を選んでいたことだ。モデルの視点から見たときに難しい例ではなく、自分にとってわかりやすいという基準で選んでいた。

改善のために実際にやったこと:

1. まず0-shot + CoTで全テストデータに対して推論させる
2. 誤分類したケースをすべて取り出す
3. 誤分類のパターンをクラスタリングする（手動で、20-30件なら意外とできる）
4. 各クラスターから代表的な例を1〜2件選んで、正しい推論プロセスを**手書きで**書く

4のステップが地味に大切だ。例の推論プロセスを自分で書くことで、「このタスクで本当に重要な判断基準は何か」を自分自身が理解できる。その理解がプロンプト全体の質を上げる副次効果もある。

あと、例の推論プロセスを書くとき、**わざと間違えて修正するプロセスを入れると効果が上がった**。「最初は請求カテゴリかと思ったが、〇〇という点から技術サポートだと判断した」という形式。これはself-correction patternと呼ばれていて、論文ベースの知見とも一致する（Kojima et al., "Large Language Models are Zero-Shot Reasoners", 2022）。

```python
# 良い例の推論プロセスの構造
GOOD_EXAMPLE_REASONING = """
分析:
1. 要約: ユーザーはアプリへのログインができないと報告している
2. 候補: 技術サポート（ログイン機能の不具合）、アカウント（アカウント停止の可能性）
3. 最初はアカウント問題かと考えたが、メール本文に「昨日まで使えていた」という記述があるため、
   アカウントではなく技術的な問題の可能性が高い。パスワードリセットの言及もないため、
   アカウント管理の問題ではなくシステム障害と判断する。
カテゴリ: 技術サポート
"""
```

## 実際に自分が推奨すること

2週間やってみた結論として、こういうシーケンスで進めることを薦める。

**Step 1: まず0-shot CoTで上限を測る**
複雑なFew-Shotを組む前に、Chain-of-Thoughtだけでどこまで行けるか確認する。シンプルなタスクなら0-shot CoTで十分なことも多い。Few-Shotを組む工数は思ったより大きいので、必要性を先に確認する。

**Step 2: Few-Shotが必要なら、例は少数・高品質で**
10〜15例で始める。例を増やすより、例の質を上げる方が効果的だった。40例より10例の方が良かったという経験は、うちでは再現性があった。

**Step 3: 例の推論プロセスは手書きで、かつ自己修正を含める**
「良い推論」を自分で書くのは面倒だけど、ここを妥協するとFew-Shotの効果が半減する。LLMに例の推論を生成させて使う方法もあるが（自己生成Few-Shot）、品質にばらつきが出るのでプロダクション用途では手書きを推奨する。

**Step 4: コスト管理を最初から設計に組み込む**
CoT + Few-Shotの組み合わせは出力トークンもプロンプトトークンも増える。Claudeのprompt cachingを使うか、バッチ処理にするか、あるいはより小さいモデル（Haiku）でCoTをやるかは、レイテンシ要件とコスト制約を見て決める。うちは最終的に、リアルタイム処理はHaiku + CoT、夜間バッチはSonnet + CoT + Few-Shotという構成にした。

---

うちのメール分類タスクは、最終的に本番で82%の精度で動いている（ヒューマンレビューとの差を埋めるため、信頼度スコアが低いものは人間にエスカレーション）。完璧ではないけど、最初の62%から考えると実用レベルに達した。

Chain-of-ThoughtとFew-Shotは確かに強力だけど、プロンプトを工夫するより、どういう例を選ぶか・どういう推論を見せるかの方が何倍も重要だった。技術よりデータの問題、という古い教訓はLLMプロンプティングでも生きていると思う。
