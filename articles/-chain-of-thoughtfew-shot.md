去年の秋、社内の契約書レビューツールを作っていたとき、GPT-4に法律文書を食わせてリスク項目を抽出させようとした。プロンプトはこんな感じ：

「この契約書のリスクを列挙してください」

結果は…まあ、ひどかった。項目がバラバラで優先度も無視、重要な免責条項を見落とすこともあった。チームリーダーから「これ使えないよ」と言われて、2週間ほどプロンプトと格闘することになった。

その格闘の記録が、今回書くことの大半だ。

## 「ゆっくり考えさせる」だけで何が変わるか — Chain-of-Thoughtの正体

Chain-of-Thought（CoT）は要するに、モデルに中間ステップを明示させる手法だ。「答えを出す前に、考え方を示してください」と指示するだけで、複雑なタスクの精度が劇的に上がることがある。

なぜ効くのか。正直、完全には理解していない。でも私の解釈はこうだ — LLMはトークンを逐次生成するので、「思考過程」を書き出すことで後続のトークン生成が前の推論に条件付けられる。つまり、モデルが自分の中間出力を文脈として利用できるようになる。

一番シンプルなCoTプロンプトのパターン：

```python
system_prompt = """
あなたは契約書のリスク分析の専門家です。
分析は必ず以下の順序で行ってください：

1. まず文書全体を読み、主要な条項を特定する
2. 各条項について、義務・権利・期間・罰則を整理する
3. 潜在的なリスクを特定し、その根拠となる条文を引用する
4. リスクを重大度（高/中/低）で分類する
5. 最後に総合的な評価を出す

# 重要: 必ず上記のステップを文章として示してから、
# 最終的な分析結果を出力すること
"""
```

このプロンプトに変えてから、契約書レビューツールの結果が明らかに改善した。特に「なぜこれがリスクか」の説明が一貫するようになったのが大きかった。

一つ注意点：CoTを使うとトークン数が増える。私のケースでは出力が平均で2〜3倍になった。コスト計算は事前にしっかりやっておいたほうがいい。

## Few-Shotの例は何でもいいわけじゃない — 選び方が全て

Few-Shot学習は「例を見せてパターンを学ばせる」手法で、基本はシンプルだ。でも最初にやらかした間違いは、**適当な例を選んでいた**こと。

初期の例はこんな感じ：

```python
examples = [
    {"input": "この契約は3年間有効です",
     "output": "リスク低：標準的な契約期間"},
    {"input": "甲は損害の一切について免責されます",
     "output": "リスク高：免責条項が広範すぎる"},
    {"input": "仲裁は東京で行う",
     "output": "リスク低：一般的な仲裁条項"},
]
```

問題は、これらの例が**エッジケースを全くカバーしていない**ことだった。「一方的な解約権」「自動更新条項」「知的財産の帰属」といった、実際の契約で頻繁に問題になる項目が含まれていなかった。

2週間かけてわかったのは、Few-Shotの例を選ぶ際には3つのことが重要だということ：

**1. タスクの難しい部分をカバーする例を選ぶ。** モデルが簡単に正解できるものじゃなく、境界線ケースや判断が難しい例を入れる。

**2. 出力フォーマットを例で示す。** 曖昧な指示より、具体的な出力形式を例で見せるほうがはるかに安定する。

**3. 例の数は多ければいいわけじゃない。** 私のテストでは、5〜7個の良質な例が、20個の粗雑な例より一貫して良い結果を出した。コンテキストウィンドウも無限じゃないし。

改善後の例には `reasoning` フィールドを追加した — これが後で重要になる：

```python
examples = [
    {
        "input": "本契約は甲の裁量により、30日前の通知をもって解約できる",
        # 思考過程を例に含めることで、CoTとFew-Shotを融合させる
        "reasoning": "解約権が甲（サービス提供者）にのみ付与されており、"
                     "乙（利用者）に同等の権利がない。一方的な条件となっている",
        "output": "リスク高：一方的解約権 — 乙は事業継続リスクを負う可能性あり"
    },
    {
        "input": "契約期間は1年とし、いずれかの当事者が3ヶ月前までに"
                 "解約の意思を示さない限り自動更新される",
        "reasoning": "自動更新は一般的だが、更新停止の通知期間（3ヶ月）が長め。"
                     "見落とすと1年の延長が確定する",
        "output": "リスク中：自動更新条項 — カレンダー管理が必要、通知期間は業界標準より長い"
    },
]
```

`reasoning` フィールドを入れた理由は次のセクションで説明する。

## CoT × Few-Shot を組み合わせると何が起きるか

実はこの2つは独立した手法じゃなくて、組み合わせると相乗効果が出る。Few-Shotの例の中にCoT的な思考過程を埋め込むことで、「このタスクではこういう風に考えるんだ」というパターンをモデルに示せる。

私が最終的に採用したプロンプト構造（Pythonで書くとこんな感じ）：

```python
def build_analysis_prompt(contract_text: str, examples: list) -> str:
    # Few-Shotの例をフォーマット
    few_shot_section = "\n".join([
        f"---例---\n入力: {ex['input']}\n思考過程: {ex['reasoning']}\n出力: {ex['output']}"
        for ex in examples
    ])
    
    return f"""
あなたは企業法務の専門家として、契約書のリスクを分析します。

以下は分析の例です：
{few_shot_section}

---
では、以下の契約条項を同じ形式で分析してください。

入力: {contract_text}
思考過程:
"""
# ↑ プロンプトを「思考過程:」で終わらせるのがポイント。
# これによりモデルは「まず推論を書く」という流れを強制される。
```

プロンプトの最後を「思考過程:」で終わらせているのが肝心で、モデルに推論ファーストで出力させる構造になっている。

うちの社内評価基準での数字（あくまで参考値、一般化はできない）：

| 手法 | 精度 |
|------|------|
| なし | 〜52% |
| Few-Shotのみ | 〜68% |
| CoTのみ | 〜71% |
| CoT + Few-Shot | 〜83% |

組み合わせの効果は数字にも出た。

## やらかしたハマりどころ3つ

### CoTが逆効果になるケース

CoTは「複雑な推論」に効く。でも単純な分類タスクや、答えが明確なルックアップ系では、むしろノイズになることがある。

別プロジェクト（ユーザーのクエリを5カテゴリに分類するタスク）にCoTを適用したとき、精度が下がった。モデルが「考える」ことで、過剰に分析しすぎて誤った分類をするケースが増えたんだ。経験則として言うと、**答えが一意に決まるシンプルなタスクにCoTは不要**。

### Few-Shotの例が偏っていた

初期の例は「リスク高」か「リスク低」の二択しかなかった。「リスク中」の例を入れていなかったせいで、モデルがほぼ全ての出力を両極端に分類してしまった。

これに気づいたのは、テスト結果を見ていたパートナーのエンジニアに「中程度のリスクって全部消えてるじゃん」と言われてから。恥ずかしい話だ（しかも2日間気づかなかった）。

### プロンプトに詰め込みすぎた

途中で欲張って「さらに判例も参照して」「業界標準との比較も」「コスト影響の試算も」と指示を追加していったら、出力が長くなりすぎて肝心な情報が埋もれるようになった。

CoT + Few-Shotは強力な分、指示を詰め込みすぎると破綻する。私の最終的なルールは「一つのプロンプトで求めるアウトプットは一種類まで」。複数の分析が必要なら、プロンプトを分けてパイプラインにする。

## 実務での使い分け — 私の判断基準

2ヶ月この手法を本番で使ってきて、判断フローはこうなっている：

**Chain-of-Thoughtを使う場合：**
- 複数ステップの推論が必要なとき
- 「なぜその答えか」の説明が必要なとき（監査ログや説明責任が求められる場面）
- 数値計算や条件評価が絡むとき

**Few-Shotを使う場合：**
- 出力フォーマットを厳密に制御したいとき
- ドメイン固有の判断基準がある（一般的な学習データには含まれにくい）
- タスクの「雰囲気」を伝えるのが難しいとき（文体、粒度感など）

**どちらも使わない場合：**
- 単純なルックアップや変換タスク
- レイテンシが最優先で、品質の多少の低下を許容できるとき

ここが肝心なんだけど、CoT + Few-Shotは「これさえ使えば全部解決する」ものじゃない。プロンプトの設計に時間がかかるし、例の質を維持するための継続的なメンテナンスも必要だ。うちのチームでは今、Few-Shotの例のデータセットをGitで管理しているけど、最初はそこまで考えていなかった（後から整理するのがかなり面倒だった）。

あと、Claude 3.5 SonnetとGPT-4oではCoTへの反応が微妙に違う。Claudeはシステムプロンプト側にコンテキストを詰める構造が相性いい印象で、GPT-4oはユーザーターン側に例を置くほうが安定した。これはモデルが変われば変わる話なので、2025年末時点での私の観察だと思ってほしい。

私のお勧めは、**まずCoTだけで始めること**。実装が簡単で効果を確認しやすい。それで不十分なら、慎重にFew-Shotを追加する。いきなり両方やると、どちらが効いているかわからなくなる。そして何より、評価データセットを作って変更ごとに数字で確認する習慣をつけること。直感や「なんとなくうまくいきそう」で進めると、必ず痛い目を見る。
