先月、うちのチーム（4人の小さなスタートアップ）で「そろそろAI APIを1本に絞ろう」という話になった。コスト管理が面倒になってきたし、ベンダーロックインをある程度許容してでも、ツールチェーンをシンプルにしたかった。というわけで、Claude・GPT-4o・Gemini 2.0を2週間ガチで使い比べた。このポストはその記録だ。

前置きとして：僕のユースケースはSaaS系のWebアプリ開発と、社内向けのドキュメント処理パイプラインの2本柱。純粋な創作系タスクはほぼやらない。そこは念頭に置いてほしい。

## コード生成：「ちゃんと動くコード」を書いてくれるのはどれか

正直、この比較を始める前は「どれもそんなに変わらないだろう」と思っていた。甘かった。

**Claude**は、コードの「意図」を読む力が一番高いと感じた。たとえば「PostgreSQLのクエリを最適化して」と雑に投げると、GPT-4oはインデックスの提案をするだけだったが、Claudeはクエリプランまで考慮した上で「このカラムの型をINTからBIGINTに変えた方がいい」という、コードの外側まで踏み込んだ提案をしてきた。コンテキストを正確に保持しながら、長いリファクタリングセッションを続けられるのも強い。

**GPT-4o**はとにかくスピードが速い。シンプルなユーティリティ関数やボイラープレートを量産するなら、これが一番ストレスが少ない。ただ、複雑なロジックになると「なんとなく動きそうだけど細部が間違ってる」コードを自信満々で出してくることがある。レビューを怠ると痛い目を見る。

**Gemini 2.0**は、Googleのエコシステムと組み合わせると強い。BigQueryのSQL、Firebase周り、GCP特有のパターン—ここは他の2つより明らかに知識が深い。ただ、それ以外の場面では「可もなく不可もなく」という印象が続いた。

実際に使ったコードを一例として載せる。各モデルに同じプロンプトを投げたときの出力品質を、簡単なテストフレームワークで計測した：

```python
import anthropic
import openai
from google import genai

# 同じタスクを3モデルに投げて結果を比較するシンプルなハーネス
# 本番用じゃなくて評価用なのでエラーハンドリングは最小限

PROMPT = """
以下のPython関数をリファクタリングしてください。
パフォーマンスとreadabilityを両立させること。

def process_items(items):
    result = []
    for item in items:
        if item['status'] == 'active':
            if item['score'] > 50:
                result.append({'id': item['id'], 'value': item['score'] * 2})
    return result
"""

def eval_claude(prompt: str) -> str:
    client = anthropic.Anthropic()
    msg = client.messages.create(
        model="claude-sonnet-4-6",  # 2026年3月時点での最新
        max_tokens=1024,
        messages=[{"role": "user", "content": prompt}]
    )
    return msg.content[0].text

def eval_gpt4o(prompt: str) -> str:
    client = openai.OpenAI()
    response = client.chat.completions.create(
        model="gpt-4o",
        messages=[{"role": "user", "content": prompt}]
    )
    return response.choices[0].message.content

def eval_gemini(prompt: str) -> str:
    client = genai.Client()
    response = client.models.generate_content(
        model="gemini-2.0-flash",
        contents=prompt
    )
    return response.text

results = {
    "claude": eval_claude(PROMPT),
    "gpt4o": eval_gpt4o(PROMPT),
    "gemini": eval_gemini(PROMPT),
}

# Claudeだけがtype hintsと、なぜその書き方が良いかの説明を自発的につけてきた
for model, output in results.items():
    print(f"=== {model} ===\n{output}\n")
```

Claudeだけが、リファクタリング後のコードに加えて「元のコードでO(n²)になりうるパターン」まで指摘してきた。頼んでないのに。これが良い意味でのお節介か、邪魔なノイズかは用途による—でも僕はこういう「一歩先の提案」が欲しい場面が多い。

**実務的な結論**：コード生成の質だけ見るならClaude。速度優先の量産タスクならGPT-4o。GCP/Firebase中心のスタックならGemini 2.0。

## 長文処理：10万トークンの壁

ここが今回一番驚いたポイントかもしれない。

うちのドキュメント処理パイプラインでは、法律文書や技術仕様書（長いものだと50〜80ページ）を要約・分類するタスクがある。全部で100KB前後のテキストを1回のリクエストに詰め込むことも珍しくない。

**Gemini 2.0**のコンテキストウィンドウは200万トークンと公称されていて、確かにそのスケールで使えた。ただ「コンテキストに入ってる ≠ ちゃんと参照できる」という問題がある。文書の後半に書いてあった重要な条件を見落とした回答が何度かあった。いわゆる「lost in the middle」問題—コンテキストが長くなるほど、中間部分への注意が薄れる現象—はGeminiでも健在だと感じた。

**Claude**は長文でも最後まで丁寧に読んでいる印象がある。80ページの仕様書から特定のエッジケースを抜き出すタスクで、文書の63ページ目に書いてあった例外条件を正確に引用してきたときは素直に感心した。コンテキストウィンドウの「実効的な使い方」が上手い。

**GPT-4o**は…ここでは正直苦手意識がある。128Kトークンまでは安定しているが、それを超えたあたりから一貫性が落ちる。うちのユースケースだと、ギリギリ足りないことが多い。

ただし注意点を一つ：Geminiの「長文はとにかく突っ込め」アプローチが合う場面もある。100ページの文書から「とりあえず全部要約して」という雑な使い方なら、Geminiが最もコスパがいい。精度より速度とコストを優先するバッチ処理なら選択肢に入る。

## 日本語の品質：エンジニアとして気になる細かい話

日本語でのやり取りを日常的にするなら、ここは無視できない。

**Claude**の日本語は自然だ。敬語の使い方が適切で、技術文書特有の硬い表現も違和感なく扱える。プロンプトを日本語で書いても、英語で書いても、出力品質にほとんど差がないのが地味に助かる（英語で考えて日本語で出力、みたいな柔軟さが効く）。

**GPT-4o**も悪くはないが、たまに「翻訳調」が滲み出ることがある。特に長い文章になると、日本語ネイティブなら選ばないような語順や語彙が出てくる。チームの日本語話者メンバーから「なんか読みにくい」と言われたことが何度かあった。

**Gemini 2.0**はGoogleの日本語データでしっかり学習している分、日常的な表現はかなり自然。ただし技術的なニュアンスになると、「意味は合ってるけど、エンジニアが実際に使う言い回しじゃない」という出力が増える。

## やらかした話：プロンプトの罠

これは恥ずかしい話なんだけど、参考になると思うから書く。

評価の初期段階で「どのモデルが一番賢いか」を測ろうとして、同じプロンプトを3つに投げて回答の「品質」を主観評価していた。2日後に気づいたんだが、Claudeへのプロンプトだけ、過去のSlackのやり取りをコンテキストとして貼り付けていた。結果として「Claudeの方が文脈理解が深い」という当たり前の結論を得ていただけだった。

評価の公平性って、思っているより難しい。特に「使い慣れているモデル」に対しては、無意識にうまいプロンプトを書いてしまう。これはどのモデル比較記事を読むときも意識しておいた方がいいと思う—書いた人がどのモデルを普段使っているかで、バイアスがかかる。

僕の場合はClaudeをメインで使っていたので、意識的に他の2つのプロンプトを丁寧に作り直してから再評価した。それでもバイアスが完全に消えているとは言い切れない。

## で、結局どれを選ぶか

ここは正直に答える。

**うちのチームはClaudeをメインAPIに選んだ。**

理由は3つ。コード生成の品質が実務の要求に一番フィットしていること。長文処理での精度が、うちのドキュメント処理パイプラインで求める水準に達していること。そして、APIのレスポンスが安定していること（GPT-4oは過去6ヶ月で2回、数時間のダウンがあった—スタートアップにとってこれは地味に痛い）。

**ただし、Gemini 2.0 Flashをサブで使い続ける。**

バッチ処理で大量の文書をざっくり分類するタスクは、コストがClaudeの約30%で済む。精度より速度とコストが優先される場面では、Flashを使うのが合理的だ。1つのモデルに全部賭けるより、ユースケースに応じて切り替える方が現実的だと判断した。

GPT-4oについては、現時点では使う理由が薄くなってしまった。価格帯はClaudeと近く、品質はうちのユースケースでは若干劣る—という評価になった。OpenAI固有のAPIエコシステム（Assistants APIとかファインチューニングとか）を活用したい場面があれば話は変わるかもしれないけど、今のプロジェクトでは不要だった。

一つ付け加えると：この選択は2026年3月時点の話だ。半年後に全部引っくり返っていても驚かない。それくらい各社のリリースペースが速い。とはいえ、「どれでも大体同じ」という時代はもう終わった。ユースケースを明確にして、自分のワークロードで実際に試すことに意味がある時代になっている。

**実務的な結論**：週に1回でも長い仕様書を読ませるならClaude。大量テキストの分類をコスト最小でやりたいならGemini Flash。OpenAIのエコシステムに深く依存しているプロジェクトならGPT-4o、それ以外での優先度は低い。
